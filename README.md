# Frontier LLM Systems

An open-source, French-language resource for understanding how frontier LLMs work and how they are run in production.

Each note is atomic, self-contained, readable in 5–10 minutes, and connected to related notes via wikilinks. The content covers model architecture, inference and serving, the engineering around the model (harness, context, structured outputs, agents), retrieval, evaluations, observability, safety, and production trade-offs.

→ Full table of contents: [`content/index.md`](content/index.md)

## Stack

The site is generated by [Quartz v5](https://quartz.jzhao.xyz/) — a static site generator built for digital gardens (wikilinks, graph view, popovers, full-text search). Content lives in Markdown under [`content/`](content/), configuration in [`quartz.config.yaml`](quartz.config.yaml).

## Contributing

This is open source by design — corrections, additions, and rewordings are welcome. The published content is in French; everything else (this README, code comments, issues, PRs) is in English.

How to help:

- **Fix an error or imprecision** — open a PR with the correction and a source link if possible.
- **Add a note** — follow the format of existing notes (H1 title, structured body, "Vocabulaire clé" section, "Synthèse" section at the end). Wire it via `[[file-name]]` wikilinks to related notes and update [`content/index.md`](content/index.md) (relevant learning path + cluster) and [`content/_vocab.md`](content/_vocab.md).
- **Improve an existing note** — concrete examples, up-to-date numbers, diagrams, references — all welcome.
- **Translate** — if you want to port a note to English or another language, open an issue first so we can agree on the structure (`.en.md` suffix, dedicated folder, etc.).
- **Report a bug or suggestion** — open an issue.

Conventions:

- One note = one concept. If you find yourself writing two independent ideas, split them.
- The canonical English term is kept as-is (KV cache, prefill, RoPE, …) because that's how it shows up in papers and provider docs.
- Minimal frontmatter — most files have none. Only add `draft: true` to exclude a note from the build.
- Run `npx prettier . --write` before committing.

Before opening a PR, run a local build (`npx quartz build`) to verify no wikilinks are broken.

## Run locally

Node 22+ required (see `.node-version`).

```bash
git clone https://github.com/AxelVincent/Frontier-LLM-Systems.git
cd Frontier-LLM-Systems
npm ci
npx quartz plugin install --from-config
npx quartz build --serve
# → http://localhost:8080, hot reload on any file change in content/
```

## Publishing your own site

The project is a static site: `npx quartz build` produces a `public/` directory that any static host can serve. Three options, from simplest to most controlled.

### Cloudflare Pages (recommended)

Generous free tier, global CDN, automatic custom domain and SSL, builds in ~1 min after the first.

1. Cloudflare Dashboard → **Workers & Pages → Create → Pages → Connect to Git**, select your fork.
2. Build settings:
   - Framework preset: **None**
   - Build command: `npm run install-plugins && npx quartz build`
   - Build output directory: `public`
   - Environment variables: `NODE_VERSION=22`
3. **Deploy**. First build ~3–5 min (plugin install), subsequent builds ~1 min.
4. (Optional) **Custom domain**: Pages → Custom domains → Set up. Cloudflare handles DNS + SSL.
5. Update `baseUrl` in `quartz.config.yaml` to your final domain, commit, push.

### Vercel

1. vercel.com → **Add New → Project → Import** your fork.
2. Build settings:
   - Framework preset: **Other**
   - Install command: `npm ci`
   - Build command: `npm run install-plugins && npx quartz build`
   - Output directory: `public`
3. **Deploy**.
4. (Optional) **Custom domain**: Settings → Domains → Add, follow Vercel's DNS instructions.
5. Update `baseUrl` in `quartz.config.yaml`, commit, push.

### GitHub Pages

Create `.github/workflows/deploy.yaml` with the workflow below, then in the repo: **Settings → Pages → Source: GitHub Actions**.

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
      - run: npm ci
      - run: npx quartz build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: public

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

If the site is served under a subpath (`<user>.github.io/<repo>`), set `baseUrl: <user>.github.io/<repo>` in `quartz.config.yaml`.

### baseUrl, in every case

`baseUrl` in `quartz.config.yaml` must point to the final domain so canonical links, sitemap, RSS, OpenGraph, and favicon resolve correctly. No scheme (`https://`), no trailing slash.

### Plugin maintenance

```bash
npx quartz plugin install --from-config   # install what the config lists
npx quartz plugin install --latest        # update to latest versions
npx quartz plugin list                    # list installed plugins
```

`quartz.lock.json` pins installed versions — commit it for reproducible builds.
